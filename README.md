## О проекте

`cryptoAppRGR` — консольное приложение для шифрования и дешифрования текста и произвольных файлов, реализованное на **C++17**. Приложение предназначено исключительно для работы на **Linux**-системах.

В рамках РГР реализованы 9 криптографических алгоритмов, объединённых в 7 динамически загружаемых плагинов (`.so`-библиотек). Главный исполняемый файл загружает нужный плагин через `dlopen`/`dlsym` в момент выбора алгоритма, что обеспечивает модульность и независимость модулей.

## Реализованные алгоритмы

### 1. AES-128 в режиме CFB

**AES (Advanced Encryption Standard)** — симметричный блочный шифр стандарта NIST с длиной ключа 128 бит и размером блока 128 бит (4×4 байта). В данном проекте используется режим **CFB (Cipher Feedback)**, при котором шифрование каждого блока зависит от предыдущего зашифрованного блока. Это превращает блочный шифр в потоковый и устраняет необходимость в дополнении до кратного размера блока.

Реализация включает полный набор раундовых преобразований: `SubBytes`, `ShiftRows`, `MixColumns`, `AddRoundKey`, а также расписание ключей (`KeyExpansion`). Ключ генерируется случайно и сохраняется в файл `aesKey.bin`.

### 2. A5/1

**A5/1** — потоковый шифр, исторически применявшийся в стандарте GSM для шифрования голосовых переговоров. Основан на трёх регистрах сдвига с линейной обратной связью (РСЛОС):

- **R1**: 19 бит, полином x¹⁹ + x¹⁸ + x¹⁷ + x¹⁴ + 1
- **R2**: 22 бита, полином x²² + x²¹ + 1
- **R3**: 23 бита, полином x²³ + x²² + x²¹ + x⁸ + 1

Итого: 64-битный ключ. На каждом такте тактируются только два из трёх регистров (по правилу большинства), что обеспечивает нелинейность. Ключ генерируется случайно и сохраняется в `a51Key.bin`.

### 3. RSA

**RSA** — асимметричный алгоритм шифрования на основе проблемы факторизации больших чисел. В данном проекте используются ключи длиной **1024 бита** (два простых числа по 512 бит). Реализация использует библиотеку **GMP (GNU Multiple Precision Arithmetic Library)** для работы с большими числами.

Этапы: генерация простых p и q → вычисление n = p·q и φ(n) = (p−1)·(q−1) → выбор открытой экспоненты e → вычисление закрытого ключа d = e⁻¹ mod φ(n). Ключевая пара сохраняется в `rsaKey.bin`. Шифрование/дешифрование выполняется поблочно с PKCS-padding.

### 4. Шифр Хилла

**Шифр Хилла** — полиграфический блочный шифр, в котором открытый текст разбивается на блоки длиной n и умножается на квадратную матрицу-ключ размером n×n по модулю 256. Для дешифрования вычисляется обратная матрица по модулю 256 (требуется, чтобы det(K) был взаимно прост с 256). Ключевая матрица генерируется из ключевого слова. Поддерживаются матрицы размером от 2×2 до 20×20.

### 5. Шифр Вернама (One-Time Pad)

**Шифр Вернама** — теоретически абсолютно стойкий потоковый шифр. Каждый байт открытого текста ксорится с соответствующим байтом случайного ключа той же длины. Ключ генерируется случайно, используется ровно один раз и сохраняется в файл. Единственный алгоритм с доказуемой абсолютной криптостойкостью при правильном использовании.

### 6. Шифр Бифид

**Шифр Бифид** — перестановочно-замещающий шифр, работающий с матрицей Полибия (5×5 для латиницы). Каждый символ открытого текста заменяется двумя координатами в матрице, после чего координаты перемешиваются и декодируются обратно. Обеспечивает эффект рассеивания (диффузии) без применения современной математики.

### 7. Шифр ADFGX

**Шифр ADFGX** — шифр, применявшийся германской армией в Первой мировой войне. Включает два этапа: сначала каждый символ заменяется двумя буквами из алфавита {A, D, F, G, X} по матрице 5×5 (дробление Полибия), затем полученный текст переставляется по ключевому слову (перестановочный шифр). Сочетает замену и транспозицию.

### 8. Аффинный шифр

**Аффинный шифр** — шифр замены, в котором каждый байт x преобразуется по формуле: `enc(x) = (a·x + b) mod 256`. Параметр `a` должен быть нечётным (взаимно простым с 256), `b` — произвольный байт. Дешифрование: `dec(y) = a⁻¹ · (y − b) mod 256`. Поддерживает шифрование текстов и файлов. Является частным случаем линейного шифра.

### 9. Шифр Плейфера

**Шифр Плейфера** — классический биграммный шифр, работающий с матрицей 16×16 (все 256 байт). Открытый текст разбивается на пары байт и шифруется согласно правилам: если байты в одной строке — сдвиг вправо; в одном столбце — сдвиг вниз; иначе — прямоугольный обмен. Матрица формируется из ключевого слова. Расширенная версия 16×16 позволяет шифровать произвольные бинарные данные.

## Архитектура проекта

Проект построен по **плагинной архитектуре**: главный бинарник (`cryptoAppRGR`) содержит только меню и загрузчик плагинов. Каждый алгоритм (или группа алгоритмов) скомпилирован в отдельную динамическую библиотеку `.so`, загружаемую в runtime через `dlopen`.

```
cryptoAppRGR (исполняемый файл)
│
│  Загружает через dlopen/dlsym
│
├── libaes.so          ← AES-128 CFB
├── liba51.so          ← A5/1
├── librsa.so          ← RSA-1024
├── libhill_vernam.so  ← Шифр Хилла + Шифр Вернама
├── libbifid_adfgx.so  ← Бифид + ADFGX
├── libaffine.so       ← Аффинный шифр
└── libplayfair.so     ← Шифр Плейфера
```

### Интерфейс плагина

Каждая `.so` экспортирует единственную функцию:

```c
void plugin_run();
```

Главное приложение ищет её через `dlsym(handle, "plugin_run")` и вызывает. Внутри `plugin_run()` расположено полное интерактивное подменю конкретного шифра. После завершения библиотека выгружается через `dlclose`.

### Поиск `.so`-плагинов

Плагины ищутся **в той же директории, что и исполняемый файл** — путь определяется через `/proc/self/exe` (специфика Linux).

### Общие компоненты

Модуль `validation.o` статически линкуется в каждый `.so`-плагин. Он предоставляет функции валидации пользовательского ввода (ввод числа, пути к файлу, ключевого слова и т.д.), которые никогда не возвращают некорректное значение и обрабатывают EOF.

### Интерфейс `crypto_interface.h`

Для аффинного шифра и шифра Плейфера используется унифицированный C-интерфейс:

```c
const AlgorithmInfo* get_algorithm_info();
size_t               get_output_size(size_t input_size, int operation_type);
int                  encrypt(ConstBuffer key, ConstBuffer input, MutBuffer* output);
int                  decrypt(ConstBuffer key, ConstBuffer input, MutBuffer* output);
```

Это обеспечивает единый паттерн вызова вне зависимости от алгоритма.

### Безопасность памяти

В `crypto_interface.h` определена функция `secure_clear_memory()` для гарантированной очистки ключевого материала из памяти после использования (препятствует оптимизациям компилятора).

## Зависимости

### Системные требования

- **ОС**: Linux (любой современный дистрибутив: Ubuntu 20.04+, Debian 11+, Fedora 36+ и др.)
- **Архитектура**: x86-64

### Необходимые пакеты

|     Пакет    |       Назначение      |       Установка (apt)         |     Установка (dnf/yum)      |
|--------------|-----------------------|-------------------------------|------------------------------|
| `g++`        | Компилятор C++17      | `sudo apt install g++`        | `sudo dnf install gcc-c++`   |
| `make`       | Система сборки        | `sudo apt install make`       | `sudo dnf install make`      |
| `libgmp-dev` | Большие числа для RSA | `sudo apt install libgmp-dev` | `sudo dnf install gmp-devel` |

Одной командой (Debian/Ubuntu):

```bash
sudo apt update && sudo apt install -y g++ make libgmp-dev
```

Одной командой (Fedora/RHEL):

```bash
sudo dnf install -y gcc-c++ make gmp-devel
```

> **Примечание:** `libdl` (динамическая загрузка) и `libstdc++fs` (работа с файловой системой C++17) входят в состав стандартной установки `g++` и отдельной установки не требуют.

## Сборка и запуск

### 1. Клонирование / распаковка

```bash
# Если проект в архиве:
unzip cryptoAppRGR.zip
cd cryptoAppRGR

# Если через git:
git clone git@github.com:Saidka-YA/S-A-I-D.git - SSH
git clone https://github.com/Saidka-YA/S-A-I-D.git - HTTPS
cd cryptoAppRGR
```

### 2. Сборка

```bash
make
```

После успешной сборки в директории проекта появятся:

```
cryptoAppRGR        ← главный исполняемый файл
libaes.so
liba51.so
librsa.so
libhill_vernam.so
libbifid_adfgx.so
libaffine.so
libplayfair.so
```

**Все `.so`-файлы должны находиться в той же директории, что и `cryptoAppRGR`.**

### 3. Запуск

```bash
./cryptoAppRGR
```

### 4. Очистка артефактов сборки

```bash
make clean
```

Удаляет все `.o`, `.so` и исполняемый файл.

## Использование

При запуске отображается главное меню с выбором группы шифров:

```
╔═════════════════════════════════════════════╗
║ 🌸      🦋КРИПТОГРАФИЧЕСКИЕ ШИФРЫ🦋      🌺 ║
╚═════════════════════════════════════════════╝

─── Выберите группу шифров ───
  1. AES-128 (CFB)
  2. A5/1
  3. RSA
  4. Хилл / Вернам
  5. Бифид / ADFGX
  6. Аффинный шифр
  7. Шифр Плейфера
  0. Выход
Выбор:
```

После выбора шифра открывается подменю конкретного алгоритма, в котором можно выбрать:

- Шифрование текста (ввод с клавиатуры) или файла
- Дешифрование текста или файла

### Работа с файлами

При выборе режима работы с файлами приложение запрашивает:

- **Путь к входному файлу** — должен существовать и быть доступен для чтения
- **Путь для сохранения результата** — директория должна существовать, файл будет создан или перезаписан

Все пути валидируются автоматически — при ошибке приложение сообщает причину и повторяет запрос.

### Управление ключами

|   Алгоритм   |          Файл ключа       |            Формат              |
|--------------|---------------------------|--------------------------------|
| AES-128      | `aesKey.bin`              | 16 байт бинарный               |
| A5/1         | `a51Key.bin`              | 8 байт бинарный                |
| RSA          | `rsaKey.bin`              | n, e, d в бинарном формате GMP |
| Шифр Хилла   | `hillKey.bin`             | Матрица n×n                    |
| Шифр Вернама | указывается пользователем | Бинарный, длина = длина текста |

Ключевые файлы создаются автоматически при шифровании и должны сохраняться для последующего дешифрования.

### Завершение работы

Для выхода из любого подменю введите `0`. Из главного меню введите `0` или нажмите `Ctrl+C`.
